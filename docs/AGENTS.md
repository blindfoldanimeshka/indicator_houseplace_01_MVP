# Правила проекта — читает каждый агент и субагент в opencode

Этот файл автоматически подхватывается opencode как базовый системный
контекст для любого агента в этой сессии (primary и subagent — без
исключений). Роль-специфичные промты лежат в `.opencode/agents/*.md`, но
всё, что написано здесь, действует поверх них всегда.

## 🔒 ОБЯЗАТЕЛЬНОЕ ПРАВИЛО: MCP-сервер Context7

**Context7 должен использоваться ВСЕГДА**, когда агент пишет, правит или
предлагает код, использующий любую библиотеку, фреймворк, API, CLI-тул
или конфиг-формат, в актуальном синтаксисе которого агент не уверен на
100% прямо в этот момент. Это касается абсолютно любого агента и
субагента в этом проекте — Python, Next.js/React/Vue, Git, CI/CD, Docker,
Frontend/Backend/DevOps, QA и Product Manager — без исключений, даже если
задача кажется тривиальной.

Обязательный порядок действий перед использованием незнакомой или часто
меняющейся библиотеки/фреймворка:
1. Вызвать `resolve-library-id` (Context7) — получить точный ID
   библиотеки/фреймворка.
2. Вызвать `query-docs` / `get-library-docs` (Context7) — получить
   актуальную документацию под нужную версию, установленную в проекте.
3. Только после этого писать или менять код.

Запрещено пропускать эти шаги на основании того, что агент "и так помнит"
синтаксис из обучающих данных — обучающие данные устаревают быстрее, чем
меняются библиотеки, а версии в этом проекте фиксируются в `package.json` /
`requirements.txt` / `pyproject.toml`, а не в памяти модели. Если Context7
недоступен (ошибка соединения) — агент обязан явно сказать об этом
пользователю в ответе, а не молча продолжать по памяти.

Context7 не нужен для: тривиальных операций с файловой системой, Git-команд
без специфичного синтаксиса, и вопросов, не связанных с конкретной
библиотекой/API.

## Язык

Общение с пользователем и комментарии в документации/PR-описаниях — на
русском. Код, имена переменных/функций, commit-сообщения — на английском
(общепринятая практика, упрощает поиск и совместимость с экосистемой
инструментов). Если это неудобно — скажите, это легко поменять.

## Общая инженерная дисциплина (проект = первый прод для соло-разработчика)

- Маленькие, атомарные изменения. Не разворачивать одну задачу в
  правку 20 файлов без необходимости.
- Перед тем как считать задачу выполненной: запустить тесты и линтер,
  которые есть в проекте, а не полагаться на то, что код "выглядит
  правильно".
- Не вводить новую инфраструктуру (очереди, отдельные сервисы, новые
  облачные ресурсы) без явного запроса — см. `docs/progress.md` (актуальный
план по фазам; историческая альтернатива Next.js — `docs/out-of-code/plan-legacy-nextjs.md`)
  проекта: цель первой версии — минимально достаточный стек, а не
  показательная архитектура.
- Секреты (токены, ключи, пароли БД) — только через переменные окружения,
  никогда не в коде и не в git-истории.
- Деструктивные операции (force-push, удаление веток, `rm -rf`, миграции
  со сбросом данных) — всегда проговорить с пользователем перед
  выполнением, даже если инструмент технически разрешает это без
  подтверждения.
- Если задача сформулирована нечётко — переспросить одним конкретным
  вопросом, а не додумывать молча и не заваливать пользователя списком
  уточнений.

## Какой агент за что отвечает

| Агент | Когда вызывать |
|---|---|
| `python` | Любой Python-код |
| `nextjs` | Next.js/React/Vite/Vue/TS/JS — фреймворковый код |
| `git` | Коммиты, ветки, merge/rebase, история |
| `cicd` | Пайплайны CI/CD (GitHub Actions и т.п.) |
| `docker` | Dockerfile, docker-compose, образы |
| `frontend-dev` | Фронтенд-задача без привязки к конкретному стеку — агент сам определяет стек по репозиторию |
| `backend-dev` | Бэкенд-задача без привязки к конкретному стеку — агент сам определяет стек по репозиторию |
| `devops-dev` | Инфраструктура/деплой без привязки к конкретному стеку |
| `qa-engineer` | Автотесты на любом языке проекта |
| `product-manager` | Ревью качества перед тем, как считать фичу готовой — вызывает `code-reviewer` и `security-reviewer` |

Подробную документацию продукта см. в файлах `README.md`, `progress.md`,
`frontend.md`, `backend.md`, `db.md`, `security-review.md`,
`technical-specification-roadmap.md` (лежат в `docs/`) — все агенты должны
сверяться с ними при работе над этим проектом, а не изобретать архитектуру
заново в каждой сессии. Актуальный индекс прогресса по фазам — `docs/progress.md`.
