# Анализ продукта «напрямую» как SaaS + бэклог фич по «болям» пользователя

> **Зачем этот файл.** У вас готов MVP (фазы 0–8 завершены, `docs/progress.md`).
> Следующий шаг — не «придумать фичи ради фичей», а осмыслить продукт как
> **SaaS-бизнес** (подписка / монетизация для бизнеса) и вывести
> конкретные фичи из **реальной головной боли пользователя**. Документ —
> основа для будущих решений «делать / не делать» (roadmap §9: scope creep
> запрещён без связи с наблюдаемой проблемой).
>
> **Статус:** аналитика + предложение. Реализация каждой фичи — отдельный
> документ в `docs/features/` и отдельная ветка. Ничего здесь не является
> указанием писать код прямо сейчас.

## Часть 1. Что у нас сейчас (facts, не мечты)

### 1.1. Продукт
Прямая аренда жилья между людьми: собственник публикует «Сдаю», арендатор
ищет «Ищу», они переписываются в чате. **Без посредников и комиссий.**
MVP уже работает на реальном стеке (React 19 + Vite + TS + Supabase).

### 1.2. Что умеет MVP (готово)
- Регистрация email+пароль, подтверждение email, reset password, legal pages.
- CRUD объявлений «Сдаю» / «Ищу» с фото (≤10, ≤10 МБ), фильтры + пагинация.
- Чат по объявлению в реальном времени (Supabase Realtime).
- Жалобы (модерация только service_role).
- Закрытая бета через инвайт-коды (`invites`, RPC-only доступ).
- 4 mock-автора + 30 seed-объявлений для демо ленты.

### 1.3. Чего НЕТ (границы MVP по roadmap §1)
- Платежи, комиссия, рейтинг, мобильное приложение.
- Автоматическая проверка документов, рекомендации, пуши.
- Публичное SEO, отдельная админ-панель.
- Капча и собственный SMTP (требуют внешних ключей — `docs/out-of-code/`).

### 1.4. Ключевая архитектурная особенность
**В MVP нет денежного потока и нет «второй стороны» бизнеса.** Все
пользователи равны (собственник ⇄ арендатор). Это важно для SaaS-фрейминга
ниже: сейчас у нас **marketplace без take-rate**, и монетизация должна
появиться поверх существующей P2P-модели, а не ломать её.

## Часть 2. Продукт как SaaS (бизнес-фрейминг)

### 2.1. Кто наш клиент (два сценария монетизации)

**Сценарий A — «B2C SaaS для листателей» (рекомендуемый первый).**
Платформа берёт деньги с **собственников/агентов**, публикующих объявления
профессионально (сдают несколько квартир, работают как мини-бизнес).
Арендаторы остаются бесплатными (как и сейчас) — это наш трафик и сеть.

- **B2C SaaS** = подписка для «профи-листателя»: тариф Free / Pro / Business.
- Ценность для клиента: больше охвата, выделение, инструменты управления
  несколькими объектами, аналитика отклика.
- Аналоги по модели: Avito/ЦИАН (частично), но без комиссии с сделки —
  доход с подписки, а не с транзакции.

**Сценарий B — «B2B SaaS для агентств/УК» (второй шаг).**
Продаём «белый ящик» или многоместный аккаунт агентству недвижимости /
управляющей компании, которой нужен свой доска объявлений для клиентов
без агрегатора-конкурента. Это продажа «инструмента», а не доступа к рынку.

### 2.2. Unit-экономика (гипотезы, проверять на бете)
- **Free**: 1 активное объявление, базовый чат. (удержание сети)
- **Pro** (~499–990 ₽/мес): до N объявлений, выделение в ленте, контакты
  вне чата (по желанию), статистика просмотров/отклика.
- **Business** (~2990–4990 ₽/мес): неограниченно, команда (несколько
  сотрудников-листателей), брендирование, API/интеграции с CRM.
- **B2B**: годовой контракт на «платформу для агентства».

### 2.3. Что меняется в ядре ради SaaS
1. **Модель аккаунта**: появляется «организация / арендодатель-бизнес»
   поверх текущего `auth.users`. Нужна таблица `organizations` +
   `organization_members` (role: owner/admin/member).
2. **Подписка**: `subscriptions` (stripe/юкаssa) + `plans` + проверка
   лимитов в RLS/триггерах (сколько объявлений может создать пользователь
   данного плана).
3. **Billing-изоляция**: платёжные ключи — только Edge Function /
   service_role, никогда в клиент (см. `docs/AGENTS.md`, правило секретов).
4. **Usage-лимиты** должны быть на стороне БД (триггер `enforce_listing_limit`
   по аналогии с `enforce_listing_image_limit`), а не только на фронте.

### 2.4. Риски SaaS-поворота (честно)
- Маркетплейс без транзакций сложнее монетизировать, чем с take-rate, но
  **безопаснее юридически** (нет агентского посредничества, проще 152-ФЗ).
- Подписка для частных листателей может отпугнуть — поэтому Free-уровень
  и бесплатные арендаторы критичны для сетевого эффекта.
- Нужен реальный платёжный провайдер (ЮKassa / Stripe) — вне-кодовая
  задача, как CAPTCHA/SMTP (`docs/out-of-code/`).

## Часть 3. «Головные боли» пользователя (source of truth для фич)

Боли выведены из природы продукта (прямая аренда, чат, фото, модерация)
и из того, чего НЕТ в MVP. Каждая боль → кандидат-фича.

### Боль 1 — «Я сдаю 5 квартир, и управлять ими больно»
**Кто:** профи-листатель (будущий плательщик Pro/Business).
**Симптомы:** нет папки объявлений, нет массовых действий, нет разделения
ролей (жена/ассистент тоже должны отвечать в чатах).
**Почему важно:** это самая монетизируемая боль — на ней строится подписка.

### Боль 2 — «Моё объявление теряется в ленте, его никто не видит»
**Кто:** любой листатель.
**Симптомы:** нет сортировки «выделенные первыми», нет статистики
(сколько просмотров/откликов), нет поднятия/обновления.
**Почему важно:** без видимости нет конверсии в чат, а без чатов нет ценности
платной подписки.

### Боль 3 — «Я боюсь скама / несуществующих квартир»
**Кто:** арендатор (бесплатный, но он — трафик).
**Симптомы:** нет верификации личности/жилья, нет рейтинга, нет
«подтверждённый собственник», легко нарваться на мошенника.
**Почему важно:** доверие — фундамент маркетплейса. Без него бета не
вырастет в публичный запуск (и 152-ФЗ-риск выше).

### Боль 4 — «Переписка в чате неудобна для сделки»
**Кто:** обе стороны.
**Симптомы:** нет шаблонов сообщений, нет прикрепления документов/договора,
нет статусов (договорились / отказ), нет уведомлений вне приложения.
**Почему важно:** чат — единственный канал связи в MVP; его качество
определяет, дойдёт ли разговор до сделки.

### Боль 5 — «Я не понимаю, где я споткнулся»
**Кто:** владелец продукта (вы) на этапе беты.
**Симптомы:** нет аналитики воронки (сколько дошли до публикации/чата),
нет событий, нет тепловой картины отвалов.
**Почему важно:** roadmap §8 (бета) требует измерять, где реальные люди
застревают — иначе фичи будут высосаны из пальца.

### Боль 6 — «Мне нужно это для команды / агентства»
**Кто:** B2B-клиент (второй сценарий).
**Симптомы:** нет многоместных аккаунтов, нет брендирования, нет API.
**Почему важно:** это путь к крупным чекам, но ТОЛЬКО после валидации B2C.

## Часть 4. Бэклог фич (каждая привязана к боли + к SaaS-ценности)

> Формат: **Фича → Боль → SaaS-ценность → Что нужно (ядро) → Сложность**.
> Приоритет: P0 (блокирует монетизацию) → P2 (nice-to-have).

### P0 — Фундамент монетизации

**F1. Организации и роли (multi-seat)**
- Боль: 1, 6. SaaS: Business-тариф.
- Ядро: `organizations`, `organization_members(role)`, привязка `listings.author_id`
  к org или к человеку; RLS на членство.
- Сложность: высокая (касается auth + RLS + чат-участников).

**F2. Тарифы и подписка (billing)**
- Боль: 1. SaaS: прямой доход.
- Ядро: `plans`, `subscriptions`, Edge Function ↔ ЮKassa/Stripe (service_role),
  webhook на обновление статуса подписки. Ключи — только secrets.
- Сложность: высокая + вне-кодовая задача (договор с провайдером).

**F3. Лимиты по тарифу (enforced server-side)**
- Боль: 1. SaaS: грань Free↔Pro.
- Ядро: триггер `enforce_listing_limit(plan)` по аналогии с
  `private.enforce_listing_image_limit()`; чтение лимита из `subscriptions`.
- Сложность: средняя.

### P1 — Видимость и конверсия (почему платят)

**F4. Продвижение и выделение**
- Боль: 2. SaaS: ценность Pro.
- Ядро: поле `promoted_until` на `listings` + сортировка ленты
  (`ORDER BY promoted_until DESC, created_at DESC`); кнопка «Поднять»
  (тратит лимит тарифа).
- Сложность: низкая/средняя (личная колонка + индекс + UI).

**F5. Аналитика объявления (views / отклики)**
- Боль: 2, 5. SaaS: аргумент для апгрейда.
- Ядро: `listing_stats` (счётчики просмотров/открытий чата), инкремент
  при открытии карточки; дашборд в профиле. Realtime-события уже есть.
- Сложность: средняя.

**F6. Уведомления (вне приложения)**
- Боль: 4. SaaS: удержание.
- Ядро: email-уведомление о новом сообщении/жалобе (Edge Function + SMTP,
  см. `docs/out-of-code/02-smtp.md`); опционально push.
- Сложность: средняя + вне-код (SMTP).

### P1 — Доверие (фундамент роста)

**F7. Верификация личности/жилья**
- Боль: 3. SaaS: снижение скама = рост сети.
- Ядро: `verifications` (тип: identity/phone/listing), бейдж «Подтверждён»
  на профиле/карточке; проверка через Edge Function (внешний провайдер
  верификации — вне-код).
- Сложность: высокая + вне-код.

**F8. Рейтинг и отзывы (после сделки)**
- Боль: 3. SaaS: сеть-эффект.
- Ядро: `reviews` (author↔tenant, по завершении чата), агрегат в `users.rating`.
  Осторожно: в MVP нет «завершения сделки» — нужен триггер статуса чата.
- Сложность: средняя/высокая.

### P2 — Удобство чата (удержание)

**F9. Шаблоны и быстрые ответы**
- Боль: 4. SaaS: экономия времени профи.
- Ядро: `message_templates` (per-user/org), UI в чате. Чисто клиент+таблица.
- Сложность: низкая.

**F10. Вложения в чат (документы/договор)**
- Боль: 4. SaaS: завершаемость сделки.
- Ядро: отдельный bucket `chat-attachments` с RLS как у `listing-photos`
  (участник чата = владелец папки); расширение `messages` типом вложения.
- Сложность: средняя.

**F11. Статусы диалога («Договорились» / «Отказ»)**
- Боль: 4. SaaS: структура вместо хаоса.
- Ядро: `chat_status` на `chats` (limited ENUM), UI-кнопки для участников.
- Сложность: низкая.

### P2 — Аналитика продукта (для вас)

**F12. Событийная аналитика воронки**
- Боль: 5. SaaS: решения на основе данных, а не интуиции.
- Ядро: Edge Function-приёмник событий (или Supabase + внешний аналитик)
  → дашборд воронки. Часть событий уже ловится Realtime.
- Сложность: средняя.

### P2 — B2B (после валидации B2C)

**F13. Брендирование и API для агентств**
- Боль: 6. SaaS: крупные чеки.
- Ядро: кастом-домен/лого для org, `api_keys` + Rate Limit. Строго после F1–F3.
- Сложность: высокая.

## Часть 5. Порядок (recommended sequence)

1. **Сначала измерить, а не строить** — F12 (аналитика воронки) на бете,
   чтобы подтвердить боли 1–5 реальными цифрами (roadmap §8, шаг 10).
2. **Потом фундамент монетизации** — F1 → F3 → F2 (нельзя брать деньги
   без организаций и лимитов; биллинг — самый тяжёлый, делать последним
   из P0, но ДО публичного запуска).
3. **Потом видимость/конверсия** — F4, F5, F6 (почему пользователь
   захочет платить Pro).
4. **Параллельно доверие** — F7, F8 (долгая история, начать рано).
5. **Потом удобство** — F9–F11 (retention, не блокирует доход).
6. **B2B только после** — F13.

## Часть 6. Что НЕ делать (explicit non-goals)
- Не вводить **комиссию с сделки (take-rate)** на старте — юридически тяжело
  (посредничество) и ломает «без посредников» позиционирование.
- Не строить **собственный платёжный шлюз** — брать готового провайдера.
- Не делать **мобильное приложение** пока веб-воронка не доказана на бете.
- Не расширять scope без записанной боли из Части 3 (scope-creep запрет,
  roadmap §9).
