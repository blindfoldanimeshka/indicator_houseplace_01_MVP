# ТЗ и roadmap MVP «напрямую»

> Статус: рабочий план реализации.  
> Дата: 19 июля 2026.  
> Цель: довести существующий кликабельный прототип до безопасного
> веб-MVP для первых 5–10 реальных пользователей.

## 1. Решение и границы

Продукт помогает людям сдавать и искать жильё без посредников. Пользователь
создаёт объявление, находит подходящее в ленте и начинает переписку с автором.

**MVP считается успешным**, если два разных зарегистрированных пользователя
могут на продовой версии:

1. зарегистрироваться и подтвердить email;
2. создать и отредактировать объявление «Сдаю» или «Ищу» с фотографиями;
3. найти активное объявление фильтрами;
4. начать диалог по чужому объявлению и обменяться сообщениями;
5. отправить жалобу, а владелец продукта может её обработать.

В MVP не входят: платежи, комиссия, рейтинг, мобильное приложение,
автоматическая проверка документов, рекомендации, пуши, публичное SEO и
отдельная админ-панель. Это сознательное ограничение, а не технический долг.

### Что уже есть и что меняем

В репозитории уже есть React/Vite-клиент, Supabase Auth, PostgreSQL, Realtime
и начальная SQL-схема. Поэтому **не нужно переписывать проект на Next.js,
NestJS и Prisma**: это увеличит срок и поверхность ошибок, не улучшив
проверку основной гипотезы. Работа начинается с аудита и укрепления текущего
решения.

Важное исключение: текущая RLS-схема разрешает любому авторизованному
пользователю создавать чат и добавлять произвольных участников. Это нельзя
считать безопасной реализацией чата. Создание чата и его участников должно
стать одной атомарной серверной операцией с проверкой автора объявления.

## 2. Целевой стек

| Слой | Выбор | Зачем |
|---|---|---|
| Клиент | React + Vite + TypeScript | Небольшое SPA; существующий интерфейс уже на этом стеке. TypeScript вводится при изменении модулей, без большого переписывания. |
| Стили | Tailwind CSS, существующие токены дизайна | Быстрая адаптивная вёрстка без отдельной дизайн-системы. |
| Данные, Auth, файлы, realtime | Supabase: PostgreSQL, Auth, Storage, Realtime | Убирает собственный сервер аутентификации, файловый сервер и WebSocket-инфраструктуру. |
| Серверная доменная операция | PostgreSQL RPC-функция для создания диалога; при необходимости одна Edge Function для действий с секретами | Критическая проверка выполняется рядом с БД и не обходится из браузера. |
| Валидация | TypeScript-типы + Zod на клиенте; `CHECK`, FK, RLS и RPC-проверки в PostgreSQL | Клиент улучшает UX, сервер остаётся единственным источником безопасности. |
| Ошибки и наблюдаемость | Sentry; логи Supabase | Узнавать о сбоях без собственной observability-платформы. |
| Тесты | Vitest + Testing Library + MSW; отдельные интеграционные проверки RLS в Supabase | Быстрые проверки интерфейса плюс проверка границы доступа. |
| Деплой | статический фронтенд на CDN/хостинге + управляемый Supabase-проект на этапе закрытой альфы | Минимум операционной нагрузки. Перед публичным РФ-запуском размещение ПДн подтверждается юридически и инфраструктурно. |

Не добавлять на MVP Redis, очереди, Docker/Kubernetes, микросервисы, Prisma
или отдельный REST-бэкенд. Они не нужны при данной нагрузке и дублируют
возможности выбранной платформы.

## 3. Архитектура и доверенные границы

```text
Браузер (React SPA, publishable key)
  ├─ Supabase Auth — сессия и подтверждение email
  ├─ Data API — только таблицы с включённым RLS
  ├─ Storage — фотографии, доступ через политики storage.objects
  └─ Realtime — только сообщения чатов, участником которых является пользователь

PostgreSQL
  ├─ таблицы и ограничения — целостность данных
  ├─ RLS — минимально необходимые права на строки
  └─ RPC open_or_create_chat — атомарное создание диалога

Привилегированная среда (не браузер)
  ├─ ключ service_role, SMTP, Sentry-токен — только secrets
  └─ Edge Function — только если необходимы секрет или внешнее API
```

`publishable`/`anon`-ключ Supabase допустимо хранить в клиентской сборке,
поскольку он не является секретом. `service_role`, строка подключения к БД,
SMTP-пароль и токены сторонних сервисов никогда не попадают в браузер, `.env`
в репозитории или логи. Защита клиентского доступа обеспечивается сочетанием
RLS и минимальных `GRANT`, а не сокрытием publishable-ключа.

## 4. Функциональное ТЗ

### 4.1 Аутентификация и профиль

- Вход и регистрация по email и паролю через Supabase Auth.
- Подтверждение email обязательно до публикации объявления или отправки
  первого сообщения. Неподтверждённый пользователь может заполнить профиль,
  но не создавать публичный контент.
- Профиль: отображаемое имя (1–100 символов), город (до 100 символов), дата
  создания. Не публиковать email, технический ID, IP или данные сессии.
- Сброс пароля через подтверждённый email; для production подключить
  собственный SMTP, так как встроенная отправка писем имеет тестовые лимиты.
- На регистрацию, вход и восстановление пароля включить CAPTCHA и настроить
  лимиты Auth. Пароль не хранится и не логируется приложением.
- Выход завершает локальную сессию; защищённые экраны проверяют её при
  загрузке и при ошибке 401/403.

Телефон и вход через Telegram не являются частью текущего MVP. Добавлять их
только после выбора провайдера, проверки стоимости и правовых требований:
они увеличивают объём персональных данных и эксплуатационные риски.

### 4.2 Объявления

Поля объявления:

| Поле | Требование |
|---|---|
| `id` | UUID, генерирует БД. |
| `author_id` | Только текущий пользователь; клиент не может назначить другого автора. |
| `type` | Ровно `offer` или `request`. |
| `city` | 2–100 символов после trim. |
| `rooms` | Значение из фиксированного справочника: студия, 1, 2, 3, 4+. |
| `price` | Целое число в рублях, больше 0, верхний бизнес-лимит фиксируется в БД. |
| `area` | Необязательное целое значение, больше 0. |
| `description` | 0–2 000 символов; отображается только как текст, без HTML. |
| `status` | `pending`, `active`, `rejected`, `archived`; на MVP публикация создаётся как `active` либо `pending` — выбрать один вариант до первого деплоя. |
| `created_at`, `updated_at`, `deleted_at` | Времена генерируются БД; удаление мягкое. |

Поведение:

- Гость видит только `active` и не удалённые объявления.
- Автор создаёт, редактирует и архивирует только свои объявления.
- Редактирование не меняет `author_id`, историю создания и статус на
  произвольный. Если выбрана ручная модерация, изменение возвращает объявление
  в `pending`.
- Лента поддерживает тип, город, комнаты, максимальную цену, сортировку
  «сначала новые» и пагинацию. Не загружать весь каталог на клиент.
- У карточки есть публичный URL/идентификатор, но контакты автора не
  раскрываются; единственный канал связи в MVP — чат.

### 4.3 Фотографии

- До 10 изображений на объявление; каждое не более 10 МБ. Разрешены JPEG,
  PNG и WebP; SVG, GIF, PDF и произвольные файлы запрещены.
- Имя объекта генерируется приложением: `listing/{listingId}/{uuid}.webp`
  либо аналогичный непредсказуемый путь. Оригинальное имя пользователя не
  используется как путь или URL.
- В отдельной таблице `listing_images` хранятся `listing_id`, `path`, порядок,
  размер, MIME-тип, дата. Нельзя считать список файлов в bucket источником
  доменных данных.
- Storage-политики разрешают загрузку и удаление только автору связанного
  объявления; лимит и MIME проверяются до загрузки. После загрузки сервер
  дополнительно проверяет сигнатуру изображения и при возможности создаёт
  безопасный WebP-превью-вариант.
- На MVP картинки объявлений могут быть публично читаемыми, но bucket не
  должен позволять listing, overwrite или upload постороннему. Если фото
  содержат чувствительные сведения, выбрать private bucket и короткоживущие
  signed URLs.

### 4.4 Чат

- Один диалог на пару «объявление + автор объявления + инициатор»;
  повторное нажатие «Написать» возвращает тот же чат.
- Автор не может открыть чат с самим собой. Нельзя начать новый чат по
  архивному, удалённому или неактивному объявлению.
- Функция `open_or_create_chat(listing_id)` в одной транзакции получает
  текущего пользователя из сессии, читает автора объявления, проверяет
  условия, создаёт чат и ровно двух участников. Прямые `INSERT` в `chats`
  и `chat_participants` из браузера запрещаются.
- Сообщение: UUID, `chat_id`, `sender_id`, текст 1–2 000 символов,
  `created_at`. Отправитель обязан быть участником чата; удалить/изменить
  сообщение в MVP нельзя.
- Список чатов доступен только участнику и не раскрывает содержание чужих
  переписок. Доставка новых сообщений — Supabase Realtime; при отказе
  соединения интерфейс делает ограниченный поллинг с backoff.
- В UI есть состояния: отправляется, отправлено, ошибка с возможностью
  повторить; запросы идемпотентны там, где повтор может создать дубль.

### 4.5 Жалобы и модерация

- Кнопка «Пожаловаться» доступна у объявления и в чате.
- Жалоба содержит тип цели, ID цели, категорию, комментарий до 1 000
  символов, автора, время и статус `new | reviewed | resolved`.
- Пользователь видит только собственную отправленную жалобу; список всех
  жалоб и смена статуса недоступны клиенту. На MVP модератор работает через
  закрытую SQL-консоль/внутренний инструмент с service role.
- Лимит: одна открытая жалоба пользователя на один объект. Действия
  модератора записываются в audit log.

## 5. Модель данных и доступы

Добавить или изменить таблицы: `listings` (статус, `updated_at`),
`listing_images`, `reports`, `moderation_audit`. Существующие `users`,
`chats`, `chat_participants`, `messages` сохраняются с усиленными
ограничениями.

Обязательные инварианты БД:

1. Не более двух участников в MVP-чате; участники различны.
2. Уникальный ключ на диалог: `listing_id + tenant_id` (где tenant — не
   автор объявления). Это устраняет дубли при двух параллельных запросах.
3. Сообщение принадлежит участнику именно того чата, в который пишется.
4. Фотография принадлежит существующему объявлению автора.
5. Каждая таблица, доступная через Data API, имеет включённый RLS,
   явные `GRANT` и политики на `SELECT`, `INSERT`, `UPDATE`, `DELETE`.

Политики должны проверяться не только по UI, но и прямыми запросами двумя
пользовательскими JWT. В частности, пользователь B не должен суметь
прочитать/изменить профиль, чат, сообщение, фото или объявление пользователя A
за исключением публичной ленты.

## 6. Нефункциональные требования

### Безопасность

- Использовать только HTTPS; задать единственный production URL и точный
  allow-list redirect URLs в настройках Auth.
- CSP, `X-Content-Type-Options: nosniff`, `Referrer-Policy`,
  `Permissions-Policy` и защита от clickjacking задаются на хостинге.
- Любые данные формы валидируются до запроса для UX и ограничениями БД/RPC
  на доверенной стороне. Не использовать пользовательские данные в HTML,
  SQL, путях файлов и фильтрах без безопасной обработки.
- Каждая SQL-функция по умолчанию `SECURITY INVOKER`. Если нужен
  `SECURITY DEFINER`, задать пустой `search_path`, полностью квалифицировать
  объекты, отозвать выполнение у `public`/`anon` и выдать его только
  `authenticated` после ревью.
- Секреты хранятся в панели провайдера или секрет-хранилище; включить
  двухфакторную аутентификацию для аккаунтов Supabase, хостинга, SMTP и Sentry.
- Зависимости обновляются регулярно, а блокирующие security alerts — до
  публичного релиза.

### Надёжность и производительность

- Страница ленты и детали объявления: целевой p95 LCP до 2,5 с на мобильной
  сети при реалистичных фото; ошибка сервера не должна ломать навигацию.
- Все запросы имеют loading, empty и error states.
- Бэкапы базы включены до появления первой боевой записи; раз в месяц
  проверяется восстановление на тестовом проекте. Восстановление не выполнять
  над production без отдельного согласованного плана.
- Ошибки клиента и серверных функций отправляются в Sentry без паролей,
  JWT, текстов чатов и иных персональных данных.

### Персональные данные и право

Проект обрабатывает как минимум email, имя, город, сообщения и фотографии;
это персональные данные. До публичного запуска для граждан РФ требуется
юридическая проверка 152-ФЗ, политика конфиденциальности, пользовательское
соглашение, согласие на обработку ПДн и утверждённые сроки хранения/удаления.
Статья 18(5) 152-ФЗ ограничивает операции при сборе ПДн граждан РФ базами за
пределами РФ; поэтому нельзя заявлять о готовности публичного российского
запуска, пока не подтверждены фактическая локация БД, Auth, Storage, бэкапов,
CDN и обработчиков ошибок. Это не юридическая консультация.

## 7. Roadmap

Работа ведётся маленькими вертикальными срезами: после каждой фазы есть
работающий деплой и проверяемый результат.

| Фаза | Результат | Работы | Критерий готовности |
|---|---|---|---|
| 0. Инвентаризация | Понятная стартовая точка | Зафиксировать состояние ветки, удалить из репозитория `.env`, сверить реальное состояние проекта с docs, завести board задач. | Нет секретов в Git; есть список расхождений и выбранная среда alpha. |
| 1. Основание | Безопасная пустая production-сборка | Настроить окружения `local`/`preview`/`production`, домен, HTTPS, переменные, Sentry, бэкапы, CI: lint + test + build. | В preview работает health-страница; CI зелёный; намеренная ошибка видна в Sentry без ПДн. |
| 2. Миграция БД и RLS | Надёжная модель доступа | Перевести `schema.sql` в последовательные миграции; добавить статусы, фото, жалобы, ограничения и RPC чата; закрыть прямые записи в чат. | Интеграционные RLS-тесты доказывают изоляцию двух пользователей и отсутствие дублей чата. |
| 3. Auth и профиль | Безопасный вход | Подтверждение email, собственный SMTP, CAPTCHA, лимиты, reset password, профиль, legal pages и consent. | Неподтверждённый аккаунт не публикует и не пишет; bot/лимит получают корректную ошибку. |
| 4. Объявления | Сквозной CRUD | TypeScript для изменяемых модулей, создание/редактирование/архив, статус, публичная лента, пагинация, фильтры, URL детали. | Автор проходит путь создать → увидеть → изменить → архивировать; другой пользователь видит только active. |
| 5. Фото | Безопасные изображения | Bucket, RLS policies, `listing_images`, лимиты, preview, удаление при архиве/удалении по политике retention. | Чужой пользователь не загружает/не удаляет файл автора; недопустимый файл отклонён. |
| 6. Чат | Связь между пользователями | RPC открытия чата, список диалогов, тред, Realtime, fallback, лимит длины и понятные ошибки. | Два аккаунта обмениваются сообщениями; третий не узнаёт о существовании чата. |
| 7. Жалобы и финальная защита | Контролируемая beta | reports, ручная модерация, audit log, security review RLS/Storage/RPC, тест бэкапа, accessibility и mobile smoke tests. | Жалоба не видна постороннему; восстановление протестировано; критических security-дефектов нет. |
| 8. Закрытая beta | Реальная обратная связь | Пригласить 5–10 людей с реальной задачей, собрать события и интервью, еженедельно устранять только блокеры. | Не менее 3 завершённых сквозных сценариев без помощи разработчика; список решений «делать/не делать» на следующую фазу. |

## 8. Очерёдность первых задач

1. Зафиксировать решение: закрытая alpha на управляемом Supabase либо
   российская инфраструктура для публичного запуска; не смешивать среды.
2. Убрать `.env` из отслеживания Git и отозвать ключи, если файл когда-либо
   попадал в удалённый репозиторий.
3. Создать миграцию, которая запрещает прямой `INSERT` в чаты/участников и
   вводит атомарную RPC-функцию.
4. Написать negative RLS-тесты для чатов, сообщений, объявлений и Storage.
5. Подключить email confirmation, SMTP, CAPTCHA и лимиты.
6. Реализовать объявления со статусом и только затем — загрузку фотографий.
7. Деплоить после каждого завершённого среза; перед beta пройти сценарий
   двумя аккаунтами в разных браузерах.

## 9. Риски и решения, которые нельзя отложить

| Риск | Решение до соответствующей фазы |
|---|---|
| Утечка чатов из-за слишком широкого RLS | Сначала закрыть прямые вставки и покрыть RLS негативными тестами. |
| Боты и фейковые объявления | Подтверждение email, CAPTCHA, лимиты, `pending` и жалобы. |
| Вредоносные или дорогие файлы | Белый список форматов, размер/количество, случайные имена, Storage RLS, серверная проверка изображения. |
| Случайная потеря данных | Бэкапы до публичных данных и проверка восстановления. |
| Невыполнение требований к ПДн РФ | Не запускать публично до юридической и инфраструктурной проверки всего маршрута данных. |
| Scope creep | Любая новая функция после beta требует связи с наблюдаемой проблемой пользователей и отдельного решения. |
