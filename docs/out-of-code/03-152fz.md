## 1. Главный риск, который текстами не закрыть

Прежде чем про Privacy/Terms — есть архитектурная проблема, которую ни один
шаблонный текст не решит:

> **Мы храним персональные данные граждан РФ (email, город, имя, в перспективе
> телефон) в Supabase, который хостится за пределами России.**

152-ФЗ (в ред. 2023 г., ст. 18) требует записи ПДн граждан РФ **только на
серверах, находящихся на территории РФ** (кроме узких исключений по
международным договорам). Supabase по умолчанию — AWS/разные регионы, **не
РФ**.

Это не «форма», а **фундаментальный вопрос выбора инфраструктуры**. `docs/plan.md`
уже рекомендует выбирать хостинг в РФ (Yandex Cloud / VK Cloud / Selectel /
Timeweb) с самого начала. `docs/progress.md` это подтверждает на уровне
философии («выбирайте хостинг в РФ с самого начала, чтобы потом не
пришлось переезжать вместе с базой пользователей»).

### Что с этим делать — три пути (обсудить с юристом + архитектором)

| Путь | Суть | Плюсы | Риски |
|---|---|---|---|
| **А. Остаться на Supabase, но вынести ПДн в РФ** | Персональные данные (email/телефон) хранить в отдельной БД на РФ-хостинге; в Supabase держать только не-ПДн (id, статусы) | Не переписывать весь стек | Архитектурная сложность, синхронизация |
| **Б. Перевести весь бэкенд в РФ** | Postgres + Auth + Storage на Yandex Cloud / VK Cloud / Timeweb | Чистое 152-ФЗ-соответствие | Переписывание Auth (Supabase → свой или РФ-аналог) |
| **В. Оставить Supabase + получить согласие на трансграничную передачу** | В Privacy явно указать, что данные обрабатываются за рубежом, получить осознанное согласие | Минимум кода | Юридически спорно для РФ-резидента; требует экспертной оценки |

**Моя техническая рекомендация:** для проекта с реальными пользователями в
РФ готовиться к **пути А или Б** и принять решение ДО публичного запуска. На
закрытой бете (фаза 8, 5–10 знакомых людей, осведомлённых тестировщиков)
риск по 152-ФЗ невелик, но **переезд базы вместе с живыми пользователями —
болезненно**, поэтому выбор инфраструктуры лучше сделать на этапе беты.

## 3. Подготовка черновиков для юриста (это можем сделать мы)

Юрист стоит денег; чтобы не платить за заполнение очевидных полей, мы
готовим **структурированный черновик**, а юрист проверяет именно правовую
часть.

### 3.1. Контуры документов (шаблон для заполнения)

**Политика конфиденциальности (Privacy)** должна покрывать:
- Какие данные собираем: имя, город, email, (позже) телефон, фото объявлений.
- Цель: регистрация, публикация объявлений, чат между пользователями.
- Правовое основание: согласие субъекта (ст. 9 152-ФЗ).
- **Где храним:** прямо указать юрисдикцию (РФ / за рубеж) — см. раздел 1.
- Передача третьим лицам: SMTP-провайдер (письма), хостинг (см. `02-smtp.md`).
- Права субъекта: запрос на удаление данных (`DELETE` из `public.users` +
  `auth.users`).
- Контакты оператора ПДн (указать ФЛ/юрлицо — **это вводит пользователь**).

**Пользовательское соглашение (Terms)** должно покрывать:
- Предмет (площадка прямой аренды без посредников).
- Обязанности пользователя (достоверность, запрет мошенничества).
- Ответственность (площадка не агент, не гарантирует сделку — важно для
  юр. защиты проекта).
- Порядок модерации и удаления объявлений (ссылка на кнопку «Пожаловаться»).
- Условия прекращения доступа.

**Согласие на обработку ПДн** — отдельный короткий текст под чекбоксом,
ссылающийся на Privacy. Уже есть в фазе 3b; оставить, но сверить формулировку
с юристом (должна быть «конкретной», а не «любые ПДн вообще» — требование
Роскомнадзора).

### 3.2. Реализация права на удаление (влияет на код)

152-ФЗ даёт пользователю право на удаление своих данных. Сейчас у нас:
- `DELETE` из `listings` (soft, через `deleted_at`) — ок.
- `public.users` удаляется каскадом при удалении `auth.users` (см. схему:
  `users.id REFERENCES auth.users(id) ON DELETE CASCADE`).

**Что сделать:** добавить в профиль кнопку «Удалить аккаунт и мои данные»,
которая вызывает `supabase.auth.admin.deleteUser` (через Edge Function с
service_role) — это каскадно снесёт профиль, его объявления и сообщения.
Это и фича, и 152-ФЗ-требование. Заложить как отдельную задачу после беты.

### 3.3. Уведомление в Роскомнадзор

Если проект становится «оператором ПДн» (а сбором email мы им становимся) —
по закону нужно **уведомить Роскомнадзор** о намерении обрабатывать ПДн.
Это делает оператор (ФЛ/юрлицо), не код. Зафиксировать как TODO для
пользователя/юриста.

5. **Внедрить правки** в Privacy/Terms + принять решение по инфраструктуре
   (раздел 1) ДО публичного запуска.
