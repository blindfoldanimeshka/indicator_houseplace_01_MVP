# ПЛАН — как дойти до прода в первый раз (LEGACY / Next.js)

Это отдельный документ поверх остальных файлов в папке. Если коротко: то,
что описано в [[frontend]], [[backend]], [[database]], [[infrastructure]] —
это зрелая архитектура «версии 2», рассчитанная на команду и на продукт,
который уже подтвердил спрос. Для **первого** реального запуска, тем более
если вы никогда раньше не доводили проект до прода, часть этого — лишняя
сложность, которая будет мешать, а не помогать. Ниже — честный, урезанный
путь, как дойти до живого продукта с реальными людьми, а не до идеальной
архитектуры на бумаге.

Прочитайте весь документ целиком один раз, прежде чем начинать что-либо
делать — порядок действий в конце имеет смысл только с учётом принципов
выше него.

## Главный принцип: сквозной путь раньше, чем полнота фич

Самая частая причина, по которой проект не доходит до прода в первый раз —
не нехватка знаний, а порядок работы: сначала месяц строится «правильный»
бэкенд, потом «правильный» фронтенд, и только в конце всё это пытаются
соединить и задеплоить — и тут вылезает тонна проблем, которые дешевле
было бы найти в первый день.

Правильный порядок — наоборот: в первую же неделю собрать максимально
тонкий, кривой, но полностью работающий путь от начала до конца —
регистрация → публикация объявления → чужой человек его видит → пишет
автору → получает ответ — и **выложить это в интернет на реальном домене**,
пусть даже страшное и без единого лишнего экрана. Дальше все фичи
добавляются поверх уже живого, задеплоенного приложения, а не
присоединяются к нему в конце.

Причина: деплой — это не финальный шаг, а самая рискованная часть проекта
для новичка. Пока вы не задеплоили хоть что-то, вы не знаете, какие
сюрпризы вас ждут (переменные окружения, домен, HTTPS, база в облаке
вместо локальной). Лучше столкнуться с этим на пустом приложении в первый
день, чем на полном приложении в последний.

## Что писать самому, а что — не изобретать

Ваш код должен решать только то, что реально уникально для вашего
продукта. Всё остальное — типовая инфраструктура, которую надёжнее взять
готовую, чем писать самому в первый раз.

**Только ваш код (это и есть продукт):**
- Логика объявлений: типы «Сдаю»/«Ищу», фильтры, лента.
- Логика чата: кто с кем переписывается по какому объявлению.
- Модель данных под это (см. [[database]] — эта часть остаётся актуальной).

**Не изобретать, а взять готовое:**
- Вход пользователей — не пишите SMS-коды, JWT, refresh-токены руками в
  первый раз. Возьмите **вход через Telegram** (Telegram Login Widget):
  пользователь жмёт «Войти через Telegram», Telegram сам подтверждает, что
  это реальный человек с реальным номером — вам не нужен SMS-провайдер,
  не нужен Redis под коды, не нужно продумывать rate limiting регистрации.
  Для рынка РФ это ещё и привычный, доверенный способ входа.
- Хранение фото — не пишите свой файловый сервер. Либо готовое объектное
  хранилище (S3-совместимое), либо для самого первого прода — просто
  папка на диске того же сервера, где крутится приложение (см. ниже про
  инфраструктуру). Это осознанное упрощение, не ошибка.
- База данных — управляемый PostgreSQL (у облачного провайдера) вместо
  того, чтобы самому администрировать сервер БД.
- Мониторинг ошибок — Sentry (бесплатного тарифа хватает на старте) вместо
  самописного логирования.

## Стек для первого прода — упрощённая версия

Вместо отдельного бэкенда на NestJS и отдельного фронтенда — **один Next.js
проект**, где серверная часть (API-роуты) и клиентская часть живут в одном
репозитории и деплоятся одной командой. Для первого раза это радикально
снижает количество мест, где что-то может сломаться: один проект, один
деплой, одни логи.

- **Next.js** — и фронтенд, и бэкенд-логика (API-роуты) в одном приложении.
- **PostgreSQL + Prisma** — как и было в [[database]], это не избыточно.
- **Telegram Login** вместо SMS-регистрации — см. выше.
- **Локальный диск сервера** (с бэкапом) вместо облачного S3 — на самом
  старте; переезд на настоящее объектное хранилище — когда фото
  накопится столько, что это станет проблемой, не раньше.
- **Обычный поллинг чата** (как в исходном прототипе — раз в несколько
  секунд), а не WebSocket/Socket.io. WebSocket — это отдельный слой
  сложности (управление соединениями, переподключения), который не нужен,
  пока не доказано, что людям вообще нужен этот чат. Апгрейд до
  реального realtime — естественный следующий шаг после MVP, не часть
  MVP.
- **Никакого Redis и очередей задач (BullMQ) на старте.** Разовые фоновые
  задачи (например, архивация старых объявлений) — обычный cron внутри
  того же сервера, без отдельной инфраструктуры очередей.
- **Никакой отдельной админ-панели.** Модерация объявлений и обработка
  жалоб на старте — это вы лично открываете таблицу в БД (у любого
  облачного PostgreSQL есть встроенный веб-интерфейс для просмотра
  таблиц) и руками меняете статус. Писать интерфейс для этого — работа,
  которая не проверяет вашу главную гипотезу.

## Что реально должно быть в коде для MVP

1. Вход через Telegram, создание пользователя при первом входе.
2. Объявления: создание (с полями из прототипа — тип, город, комнаты,
   цена, площадь, описание), загрузка фото, лента с фильтрами, просмотр
   одного объявления, редактирование/удаление своего.
3. Поле статуса объявления (`pending`/`active`/`rejected`) в базе данных —
   даже если решение о смене статуса вы принимаете вручную через таблицу
   в БД, само поле должно быть в коде с первого дня. Добавить его потом,
   когда уже есть боевые объявления в базе — намного дороже.
4. Чат: начать переписку из объявления, отправить/получить сообщение,
   список своих диалогов, поллинг для обновления.
5. Кнопка «Пожаловаться» — на первом этапе достаточно, чтобы она
   открывала письмо вам на почту с контекстом (какое объявление/диалог),
   а не заводила отдельную систему жалоб.

Это весь список. Если тянет добавить что-то сверх — сначала проверьте
по разделу ниже, не из «сознательно не сейчас» ли это.

## Что сознательно не строим сейчас

- Верификацию собственника через документы — сначала нужно узнать, готовы
  ли люди вообще пользоваться площадкой; верификация решает проблему
  доверия, которая имеет смысл только при наличии живых объявлений и
  живых пользователей.
- Отдельные роли «арендодатель»/«арендатор» — любой пользователь может
  публиковать любой тип объявления, как и в прототипе.
- WebSocket-чат, push-уведомления, email-рассылки.
- Платное продвижение объявлений и любая монетизация.
- Мобильное приложение — адаптивная вёрстка в браузере закрывает это на
  старте.
- SEO-оптимизация страниц — важна, когда уже есть что индексировать
  (реальные объявления), а не в момент, когда лента пустая.

Каждый пункт из этого списка — не «никогда», а «не раньше, чем появятся
живые пользователи и станет понятно, что именно из этого действительно
нужно».

## Порядок действий по шагам

1. **Поднять пустой скелет и задеплоить его.** Next.js + PostgreSQL,
   одна страница-заглушка, которая читает что-то из базы и показывает на
   экране. Задеплоить на реальный сервер/домен до того, как написана хоть
   одна настоящая фича. Цель шага — один раз пройти весь путь «код на моём
   компьютере → код доступен по ссылке в интернете» и разобраться со
   всеми сюрпризами (переменные окружения, домен, HTTPS) на пустом месте.
2. **Вход через Telegram**, полностью — от кнопки «Войти» до того, что
   пользователь создаётся в базе и его видно в таблице `users`.
   Задеплоить.
3. **Объявления от начала до конца**: форма создания → запись в БД →
   отображение в ленте. Без фото и без фильтров — просто голый CRUD.
   Задеплоить, попробовать на себе.
4. **Фото объявлений.**
5. **Фильтры в ленте** (город, тип, комнаты, цена).
6. **Чат на поллинге**: начать переписку из объявления, отправить и
   получить сообщение, список диалогов.
7. **Кнопка «Пожаловаться»** (см. выше — просто письмо вам).
8. Пройти весь путь самостоятельно два раза подряд, под двумя разными
   аккаунтами (два браузера/два Telegram-аккаунта), играя за
   «собственника» и за «арендатора» — до первого реального пользователя
   баги должны найти вы, а не он.
9. **Позвать 5–10 живых людей**, которым реально нужно снять или сдать
   жильё — не абстрактных тестировщиков, а тех, у кого есть настоящая
   потребность прямо сейчас. Дать им ссылку без объяснений «как этим
   пользоваться» и смотреть, где они спотыкаются.
10. Только после этого шага решать, что добавлять дальше — по тому, что
    реально мешало живым людям, а не по списку красивых фич.

## Что значит «готово к проду» лично для вас на этом этапе

Продакшн-готовность — это не «энтерпрайз-уровень», а отсутствие вещей,
которые реально могут вас подвести:

- Секреты (токен Telegram-бота, пароль от БД) — только в переменных
  окружения сервера, никогда не в git-репозитории.
- HTTPS на домене (бесплатный сертификат Let's Encrypt — почти любой
  современный обратный прокси настраивает это автоматически).
- Резервные копии базы данных — если у облачного провайдера они
  включаются одной галочкой, включите её в первый же день, не
  откладывайте на «потом».
- Хоть какая-то видимость ошибок в проде — бесплатного Sentry достаточно,
  чтобы не узнавать о падении сайта от пользователя в комментариях.
- Ничего из пользовательского ввода не должно напрямую попадать в SQL-
  запросы или в HTML без экранирования — Prisma и React дают это по
  умолчанию, просто не обходите их «ради скорости».

Это весь минимум. Kubernetes, CI/CD-конвейеры, incident response —
осмысленны, когда есть команда и реальная нагрузка, не на старте с одним
разработчиком.

## Про 152-ФЗ и юридическую часть — честно

Вы будете хранить телефон и, в перспективе, документы пользователей —
персональные данные граждан РФ, которые по закону должны обрабатываться
на серверах в России. Это реальное требование, а не формальность, и я не
юрист — при первом настоящем публичном запуске стоит свериться с
юристом или хотя бы внимательно почитать актуальные требования
Роскомнадзора самому.

Практическое разделение по этапам:
- Пока это закрытая проверка на 5–10 знакомых людях, которые в курсе, что
  тестируют ваш прототип — риск невелик, но выбирайте хостинг в РФ
  (Yandex Cloud / VK Cloud / Selectel / Timeweb) с самого начала, чтобы
  потом не пришлось переезжать вместе с базой пользователей.
- Перед тем как звать незнакомых людей публично — обязательны
  пользовательское соглашение, политика конфиденциальности и согласие на
  обработку персональных данных при регистрации. Это буквально текстовые
  страницы + один чекбокс при регистрации, не тянет отдельной фазы
  разработки, но забыть про это нельзя.

## Типичные ошибки новичка, которые чаще всего мешают дойти до конца

- Строить «правильную» инфраструктуру (очереди, микросервисы, admin-панели)
  до того, как появился хоть один живой пользователь.
- Откладывать первый деплой до момента, когда «уже почти всё готово» —
  вместо того чтобы деплоить с первого дня и добавлять фичи в уже живое
  приложение.
- Пытаться сразу реализовать всё из [[roadmap]] и [[backend]] — те файлы
  описывают горизонт на будущее, а не стартовую точку.
- Молчаливо считать, что раз код написан — значит, работает. Единственная
  проверка — вы сами прошли весь путь целиком в задеплоенной версии, а
  затем это сделал кто-то, кроме вас.
