# 02 — Собственный SMTP для production-рассылки

> **Статус в проекте:** 🔲 Решение за пользователем. По `docs/progress.md`
> встроенная отправка Supabase допустима для закрытой беты, но по ТЗ §4.1
> для production нужен выделенный SMTP. Этот документ — план выбора
> провайдера, настройки и влияния на существующий `email-confirmation guard`.

---

## 1. Почему встроенного SMTP Supabase недостаточно

Supabase Auth «из коробки» отправляет письма (подтверждение email, сброс
пароля) через **свой домен отправителя**, который:

- имеет **очень низкий rate limit** (демонстрационный, несколько писем/час);
- часто попадает в **спам** у крупных почтовых сервисов;
- использует домен `*.supabase.co` — получатель видит «от кого-то ещё»,
  доверия нет;
- для РФ-проекта данные прохождения письма (заголовки, метаданные) уходят
  за рубеж — косвенный 152-ФЗ-риск (см. `03-152fz.md`).

Из документации Supabase: *"It is strongly recommended to configure a custom
SMTP provider for Supabase projects to avoid issues with sending emails. The
built-in email provider is intended for demonstration purposes only and has a
very low rate limit, making it unsuitable for production environments."*

Для нашего проекта это критично по двум причинам:

1. **`email-confirmation guard` (фаза 3):** пользователь НЕ попадает в
   приложение, пока email не подтверждён. Если письмо подтверждения не
   доходит → пользователь застрял на экране регистрации. На закрытой бете
   это блокирует тестировщиков.
2. **`resetPasswordForEmail` (фаза 3b):** без рабочего SMTP сброс пароля
   не работает вообще.

## 2. Выбор провайдера (фокус: РФ, 152-ФЗ)

Supabase Auth совместим с **любым SMTP-сервисом**. Для российского проекта
приоритет — провайдеры с инфраструктурой в РФ (данные писем не уходят
за рубеж) и поддержкой SPF/DKIM.

| Провайдер | Инфра в РФ | Для чего | Примечание |
|---|---|---|---|
| **Unisender** (UniOne) | ✅ Да | Транзакционная почта (API+SMTP) | Ориентирован на РФ, хорошая доставляемость |
| **Mailopost** | ✅ Да | SMTP-рассылка | РФ-резидент, дешево на старте |
| **Sendsay** | ✅ Да | Транзакционные письма | РФ, корпоративный уровень |
| **Selectel / Timeweb** (SMTP-услуги) | ✅ Да | Хостинг+почта в пакете | Удобно, если там же БД |
| ~~Resend~~ | ❌ США | — | Отличный API, но за рубежом — 152-ФЗ минус |
| ~~Amazon SES~~ | ❌ США | — | Дешево, но модерация сложна из РФ |
| ~~SendGrid~~ | ❌ США | — | То же |

### Рекомендация

**Unisender (тариф UniOne)** для старта: это транзакционный SMTP (именно
то, что нужно для писем подтверждения и сброса пароля, а не маркетинговая
рассылка), инфраструктура в РФ, понятная панель, есть SMTP-доступ в базовом
тарифе. Бесплатный лимит позволяет прогнать закрытую бету бесплатно.

> Если уже арендуете сервер у Selectel/Timeweb — можно взять их SMTP в том
> же аккаунте, чтобы сократить количество сервисов.

## 3. Настройка в Supabase Auth

Два способа — через Dashboard (проще) или Management API (воспроизводимо,
хорошо для документации).

### Способ А — Dashboard

Проект `MVP-House` → **Authentication → Settings → SMTP Settings**:

| Поле | Значение |
|---|---|
| Enable Custom SMTP | ✅ Вкл |
| SMTP Host | `smtp.unisender.com` (уточнить у провайдера) |
| SMTP Port | `587` (STARTTLS) или `465` (SSL) |
| SMTP User | логин из панели провайдера |
| SMTP Password | пароль из панели провайдера |
| Sender email | `no-reply@ваш-домен.ru` (домен — ваш, не провайдера!) |
| Sender name | «напрямую» (или «Прямая аренда») |

**Важно:** поле `Sender email` должно быть на **вашем собственном домене**,
а не на домене провайдера. Для этого домен должен быть подтверждён и на нём
настроены SPF/DKIM (раздел 4).

### Способ Б — Management API (рекомендую фиксировать в README проекта)

Токен и project-ref — в переменных окружения, НЕ в git.

```bash
export SUPABASE_ACCESS_TOKEN="ваш-токен"     # https://supabase.com/dashboard/account/tokens
export PROJECT_REF="ваш-project-ref"

curl -X PATCH "https://api.supabase.com/v1/projects/$PROJECT_REF/config/auth" \
  -H "Authorization: Bearer $SUPABASE_ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "external_email_enabled": true,
    "mailer_secure_email_change_enabled": true,
    "mailer_autoconfirm": false,
    "smtp_host": "smtp.unisender.com",
    "smtp_port": 587,
    "smtp_user": "ваш-smtp-user",
    "smtp_pass": "ваш-smtp-password",
    "smtp_admin_email": "no-reply@ваш-домен.ru",
    "smtp_sender_name": "напрямую"
  }'
```

> Параметр `mailer_autoconfirm: false` — это и есть наш `email-confirmation
> guard` на уровне сервера: пользователь обязан подтвердить почту. Не
> выставляйте `true` даже на бете — иначе guard теряет смысл и спам-аккаунты
> сразу активны.

## 4. DNS-записи (ОБЯЗАТЕЛЬНО — иначе письма в спам)

Это самая частая причина «письма не доходят». Нужно прописать в панели
вашего DNS-провайдера (где куплен домен) ТРИ записи для `ваш-домен.ru`:

### SPF (TXT)

```
ваш-домен.ru.  IN TXT  "v=spf1 include:spf.unisender.com ~all"
```
(точное значение `include:` даёт провайдер в документации — скопируйте
буквально. Несколько провайдеров → перечислить через пробел в одной TXT.)

### DKIM (TXT)

Провайдер даёт селектор и значение. Пример (скопируйте реальное у
провайдера, не используйте этот плейсхолдер!):

```
s1._domainkey.ваш-домен.ru.  IN TXT  "v=DKIM1; k=rsa; p=MIGfMA0GCSqGSIb..."
```

### DMARC (TXT) — рекомендуется

```
_dmarc.ваш-домен.ru.  IN TXT  "v=DMARC1; p=none; rua=mailto:postmaster@ваш-домен.ru"
```
После прогрева домена `p=none` → `p=quarantine` → `p=reject`.

### Как проверить

Через пару часов после изменений DNS:

```bash
dig +short txt ваш-домен.ru
dig +short txt _dmarc.ваш-домен.ru
```
И онлайн-чекеры вроде `mail-tester.com` (отправить тестовое письмо через
Supabase Auth → проверить оценку; цель ≥ 9/10).

## 5. Влияние на существующий код

### email-confirmation guard (фаза 3)

В `AuthProvider` есть клиентская проверка: пользователь с неподтверждённым
email не допускается в приложение. **С SMTP это начинает реально работать:**

- Было (встроенный SMTP): письмо часто не доходит → пользователь заблокирован
  на экране регистрации, не понимая почему.
- Стало (ваш SMTP + SPF/DKIM): письмо доходит → пользователь подтверждает →
  guard пропускает.

**Что сделать в коде (минимум):** на экране ожидания подтверждения добавить
понятный русский текст «Письмо отправлено на {email}. Проверьте папку
«Спам». Не пришло? [Отправить повторно]» и кнопку повторной отправки через
`supabase.auth.resend()`. Без этого UX на бете будет страдать.

### resetPasswordForEmail (фаза 3b)

Заработает автоматически после настройки SMTP — отдельный код не нужен.
Проверить вручную на бете: форма сброса → письмо приходит → ссылка ведёт на
страницу обновления пароля.

### Уведомления о жалобах (фаза 7)

Кнопка «Пожаловаться» открывает письмо модератору. Если это делается через
`mailto:` — SMTP не нужен. Если вы захотите отправлять жалобы
программно (Edge Function) — используйте тот же SMTP (отдельный отправитель
`moderation@ваш-домен.ru`).

## 7. Частые ошибки

| Ошибка | Последствие | Как избежать |
|---|---|---|
| `mailer_autoconfirm: true` | Спам-аккаунты сразу активны | Держать `false` |
| Sender email на домене провайдера | Письма в спам, низкое доверие | Только ваш домен |
| Нет SPF/DKIM | Доставляемость ~0% | Обязательно прописать DNS |
| SMTP-пароль в `.env` проекта | Утечка через git | Только в Supabase config |
| Оставить встроенный SMTP на prod | Лимиты/спам, guard «молчит» | Переключить до публичного запуска |
| Забыть про кнопку ресенда | Пользователь застрял | Добавить в экран ожидания |
